Design doc for Micro-kernel project describing decisions, challenges, and tradeoffs faced

ISA (Instruction Set Architecture):

For this project I chose 32-bit RISC-V for the ISA.
I chose 32-bit over 64-bit as its simplier to read.
I chose RISC-V over ARM or x86 as its simplier, and has lots of use in embedded devices,
and thats where my interests lie as a firmware engineer.

Micro-Kernel Features:
Implemented:
- RV32 kernel boot path in S-mode under OpenSBI.
- Console I/O through SBI calls (`putchar`, `getchar` via `ecall`).
- `.bss` zeroing during early boot.
- Basic common library routines: `memset`, `memcpy`, `strcpy`, `strcmp`.
- Minimal `printf` support for `%d`, `%x`, `%s`, and `%%`.
- Trap entry/exit path (`kernel_entry`) that saves/restores all general-purpose registers and returns with `sret`.
- Trap frame definition and syscall dispatch (`handle_trap`, `handle_syscall`).
- System call ABI: SYS_EXIT, SYS_PUTCHAR, SYS_GETCHAR, SYS_READFILE, SYS_WRITEFILE.
- Early physical page allocator (`alloc_pages`) using linker-defined free RAM region.
- SV32 two-level page table mapping (`map_page`).
- Process control structure, round-robin scheduler, and context switching (`create_process`, `yield`, `switch_context`).
- User-mode execution: processes launched via `sret` into U-mode at USER_BASE (0x1000000).
- Per-process page tables with kernel memory mapped 1:1 and user image mapped at USER_BASE.
- Virtio-blk device driver: virtqueue initialization, descriptor chain construction, polling completion.
- TAR-based in-memory filesystem with read and write persistence to virtio-blk disk.
- Interactive command-line shell running in user mode (hello, readfile, writefile, exit).
- Build and run flow through `run.sh` using clang/lld and QEMU; all artifacts go to `build/`.

Not Yet Implemented:
- Timer interrupts and preemption (currently cooperative scheduling only).
- Inter-process communication.
- Dynamic process creation from user space.
- Interrupt-driven I/O (currently polling-based for both console and disk).
- Page reclamation, demand paging, or swap.
- Multi-core support.

Source code structure:
├── build/         - All build artifacts (*.elf, *.bin, *.map, disk.tar, qemu.log)
├── disk/          - Filesystem contents (packed into build/disk.tar at build time)
├── docs/          - Design notes and reference material
├── common.c       - Kernel/user shared library: printf, memset, memcpy, strcpy, strcmp
├── common.h       - Kernel/user shared library: types, PAGE_SIZE, syscall numbers
├── kernel.h       - Kernel: cross-cutting types (process, trap_frame), PANIC/CSR macros
├── kernel.ld      - Kernel: linker script (memory layout)
├── sbi.c / .h     - SBI console interface: sbi_call, putchar, getchar
├── mm.c / .h      - Memory management: alloc_pages, map_page (SV32 paging)
├── trap.c / .h    - Trap handling: kernel_entry (asm), handle_trap, handle_syscall
├── proc.c / .h    - Process management: create_process, yield, switch_context
├── virtio.c / .h  - Virtio-blk driver: virtio_blk_init, read_write_disk
├── fs.c / .h      - Filesystem: fs_init, fs_lookup, fs_flush (TAR format)
├── main.c         - Kernel entry point: kernel_main, boot
├── shell.c        - User-space interactive shell
├── user.c / .h    - User-space syscall wrappers
├── user.ld        - User: linker script
└── run.sh         - Build script

CPU modes:
Mode	Overview
M-mode	Mode in which OpenSBI (i.e. BIOS) operates
S-mode	Mode in which the kernel operates, aka. "kernel mode"
U-mode	Mode in which applications operate, aka. "user mode"

Privileged instructions:
Opcode and operands	    Overview	                            Pseudocode
csrr rd, csr	        Read from CSR                           rd = csr;
csrw csr, rs	        Write to CSR	                        csr = rs;
csrrw rd, csr, rs	    Read from and write to CSR at once	    tmp = csr; csr = rs; rd = tmp;
sret	                Return from trap handler
sfence.vma	            Clear Translation Lookaside Buffer (TLB)

File Sections:
.text	This section contains the code of the program.
.rodata	This section contains constant data that is read-only.
.data	This section contains read/write data.
.bss	This section contains read/write data with an initial value of zero.

3 Privilege Levels:
Machine Mode    (M-mode): The highest privilege level, where code has unrestricted access to all hardware resources (memory, registers, etc.).
                          It is responsible for booting, handling low-level exceptions, and is mandatory on all RISC-V harts.
Supervisor Mode (S-mode): Designed for modern operating systems, providing privilege separation between the operating system kernel and applications.
                          It is supported in systems running systems like Linux.
User Mode       (U-mode): The lowest privilege level for running application code.
                          Code in U-mode has restricted access to hardware, protecting the system from accidental or malicious interference.

Current `printf` support: `%d` (decimal), `%x` (hexadecimal), `%s` (string), and `%%`.
