Design doc for Micro-kernel project describing decisions, challenges, and tradeoffs faced

ISA (Instruction Set Architecture):

For this project I chose 32-bit RISC-V for the ISA.
I chose 32-bit over 64-bit as its simplier to read.
I chose RISC-V over ARM or x86 as its simplier, and has lots of use in embedded devices, 
and thats where my interests lie as a firmware engineer.

Micro-Kernel Features:
Implemented:
- RV32 kernel boot path in S-mode under OpenSBI.
- Console output through SBI calls (`putchar` via `ecall`).
- `.bss` zeroing during early boot.
- Basic common library routines: `memset`, `memcpy`, `strcpy`, `strcmp`.
- Minimal `printf` support for `%d`, `%x`, `%s`, and `%%`.
- Trap entry/exit path (`kernel_entry`) that saves/restores general-purpose registers and returns with `sret`.
- Trap frame definition and panic-based trap handling (`handle_trap`).
- Early physical page allocator (`alloc_pages`) using linker-defined free RAM region.
- Linker-defined memory layout with kernel image, stack, and free RAM region.
- Build and run flow through `run.sh` using clang/lld and QEMU.

Not Yet Implemented:
- Scheduler and multitasking/process switching.
- Running user programs in U-mode.
- User-space support library and linker script (`user.c`, `user.h`, `user.ld` are currently empty).
- System call ABI and syscall dispatching.
- Structured exception/interrupt handling (currently unexpected traps panic).
- Timer interrupts and preemption.
- Virtual memory and paging (`satp`, page tables, `sfence.vma`).
- Device drivers beyond SBI console output.
- File system implementation and use of `disk/` contents.
- Interactive shell (`shell.c` is currently empty).

CPU modes:
Mode	Overview
M-mode	Mode in which OpenSBI (i.e. BIOS) operates
S-mode	Mode in which the kernel operates, aka. "kernel mode"
U-mode	Mode in which applications operate, aka. "user mode"

Privileged instructions:
Opcode and operands	    Overview	                            Pseudocode
csrr rd, csr	        Read from CSR                           rd = csr;
csrw csr, rs	        Write to CSR	                        csr = rs;
csrrw rd, csr, rs	    Read from and write to CSR at once	    tmp = csr; csr = rs; rd = tmp;
sret	                Return from trap handler	
sfence.vma	            Clear Translation Lookaside Buffer (TLB)

Source code structure:
├── disk/     - File system contents
├── common.c  - Kernel/user common library: printf, memset, ...
├── common.h  - Kernel/user common library: definitions of structs and constants
├── kernel.c  - Kernel: process management, system calls, device drivers, file system
├── kernel.h  - Kernel: definitions of structs and constants
├── kernel.ld - Kernel: linker script (memory layout definition)
├── shell.c   - Command-line shell
├── user.c    - User library: functions for system calls
├── user.h    - User library: definitions of structs and constants
├── user.ld   - User: linker script (memory layout definition)
└── run.sh    - Build script

File Sections:
.text	This section contains the code of the program.
.rodata	This section contains constant data that is read-only.
.data	This section contains read/write data.
.bss	This section contains read/write data with an initial value of zero.

3 Privilege Levels:
Machine Mode    (M-mode): The highest privilege level, where code has unrestricted access to all hardware resources (memory, registers, etc.).
                          It is responsible for booting, handling low-level exceptions, and is mandatory on all RISC-V harts.
Supervisor Mode (S-mode): Designed for modern operating systems, providing privilege separation between the operating system kernel and applications.
                          It is supported in systems running systems like Linux.
User Mode       (U-mode): The lowest privilege level for running application code.
                          Code in U-mode has restricted access to hardware, protecting the system from accidental or malicious interference.

Current `printf` support: `%d` (decimal), `%x` (hexadecimal), `%s` (string), and `%%`.
